# Inverse Hunter - ゲーム改善設計書

## 1. はじめに

本設計書は、ユーザーから提供されたゲームデザインドキュメント（GDD）と既存のコードベース（`index.html`）を基に、3DアクションRPG「Inverse Hunter」のゲーム性、操作性、ゲームバランスを原神やゼルダレベルに大幅に改善するための設計方針を記述するものです。現在の「完成度の低さ、アクション性、デザイン、操作性の悪さ」といった課題を解決し、ユーザーが求める高品質なゲーム体験を実現することを目的とします。

## 2. 現状分析と課題

### 2.1. 既存コードベースの評価

現在のゲームは、単一の`index.html`ファイル内にHTML、CSS、JavaScriptが混在する形で実装されています。Three.jsを用いて3Dレンダリングが行われており、地形、プロップ、クリーチャーの生成、基本的なプレイヤー入力、カメラ制御、UI要素などが実装されています。サウンドはWebAudio APIによる合成音源が使用されています。

**良い点:**
*   Three.jsによる基本的な3D環境が構築されている。
*   GDDに記載された変身システム、本能ゲージ、感覚システム、生態系AI、仲間システム、戦闘設計、ナラティブの概念がコードの随所に散見される。
*   PC（キーボード/マウス）とモバイル（ジョイスティック/ボタン）の両方の入力に対応しようとしている。
*   プロシージャル生成による地形やクリーチャーの試みがなされている。

**課題点:**
*   **単一ファイル構造:** コードの可読性、保守性、拡張性が極めて低い。大規模なゲーム開発には不向き。
*   **モジュール化の欠如:** 各機能が密結合しており、特定の機能の変更が他の部分に影響を与えやすい。
*   **アセット管理:** 3Dモデル、テクスチャ、アニメーションなどの外部アセットの管理が未整備。高品質なグラフィックを実現するためには必須。
*   **ゲームロジックの複雑性:** プレイヤー、敵、ワールド、UIなどのロジックが混在し、デバッグや機能追加が困難。
*   **パフォーマンス:** 大規模なワールドや多数のオブジェクトを扱う際のパフォーマンス最適化が不足している可能性。
*   **バージョン管理:** GitHubへの逐次プッシュは行われているものの、単一ファイルでの変更はコンフリクトや履歴管理の面で非効率。

### 2.2. ユーザーからのフィードバックとGDDからの課題抽出

ユーザーからのフィードバックとGDDを総合すると、以下の点が主要な改善点として挙げられます。

*   **ゲーム性:**
    *   「完成度低い」「アクション性もデザインも操作性も悪い」という全体的な評価。
    *   「原神やゼルダレベルのガチ3DRPGゲーム」への大幅改造。
    *   GDDの「変身チェーン」「本能ゲージ」「感覚システム」「生態系AI」「仲間システム」「弱点×形態の掛け算バトル」「記憶の断片」「プレイヤー感情曲線」といったユニークなコンセプトを深く掘り下げ、ゲームシステムとして機能させる必要がある。
*   **操作性:**
    *   「キーボードの矢印があべこべになってる」という明確なバグの修正。
    *   「攻撃もエイムできるようにしたりして」という要望。
    *   PC/スマホ横画面への対応の強化。
*   **ゲームバランス:**
    *   全体的なゲームバランスの調整。
    *   GDDの「弱者が連鎖を昇り、やがて守る者になる」「強くなることの意味」「守る重さ」といったテーマをゲームシステムに落とし込むバランス調整。

## 3. 改善設計方針

### 3.1. アーキテクチャ設計

「原神やゼルダレベル」のゲームを目指す上で、現在の単一ファイル構造は根本的に変更する必要があります。以下に新しいアーキテクチャの提案を示します。

**3.1.1. モジュール化とコンポーネント指向**

*   **JavaScriptモジュール:** すべてのJavaScriptコードをESモジュール（`.js`ファイル）に分割し、機能ごとにファイルを整理します。これにより、コードの再利用性、可読性、保守性を向上させます。
    *   `src/` ディレクトリ以下に、`core/` (エンジンコア、ユーティリティ)、`game/` (ゲームロジック、プレイヤー、敵、AI)、`world/` (地形、プロップ、環境)、`ui/` (HUD、メニュー)、`assets/` (アセットローダー、マネージャー)、`input/` (入力処理) などのサブディレクトリを作成します。
*   **コンポーネント指向:** プレイヤー、敵、プロップ、UI要素などを独立したコンポーネントとして設計し、それぞれの振る舞いやデータをカプセル化します。これにより、システムの複雑性を管理しやすくします。
*   **状態管理:** ゲームの状態（プレイヤーHP、インベントリ、ステージ進行など）を一元的に管理するメカニズムを導入します。Reduxのようなパターンや、シンプルなグローバル`GAME`オブジェクトの拡張を検討します。

**3.1.2. アセットパイプラインの導入**

*   **3Dモデル:** BlenderなどのDCCツールで作成されたGLTF/GLB形式の3DモデルをThree.jsでロード・表示できるようにします。各変身形態の高品質なモデル、敵キャラクター、環境オブジェクトなど。
*   **テクスチャ:** PBR（Physically Based Rendering）に対応したテクスチャ（Albedo, Normal, Roughness, Metalnessなど）を使用し、リアルなマテリアル表現を目指します。
*   **アニメーション:** 3Dモデルに紐づくアニメーション（歩行、攻撃、変身など）を実装します。Three.jsのAnimationMixerを活用します。
*   **サウンド:** WebAudio APIによる合成音源だけでなく、高品質な効果音（SFX）やBGMを外部ファイル（MP3/OGG）としてロードし、管理するシステムを導入します。

**3.1.3. ビルドシステム**

*   **バンドラー:** WebpackまたはRollupを導入し、複数のJavaScriptモジュール、CSS、アセットを一つのHTMLファイルに統合するためのビルドプロセスを構築します。これにより、デプロイメントが容易になり、ロード時間の最適化も可能になります。
*   **開発サーバー:** 開発効率向上のため、ホットリロード機能を備えた開発サーバーを導入します。

### 3.2. ゲームシステム設計

GDDのコンセプトを深く掘り下げ、原神やゼルダのような体験を提供するための具体的なゲームシステム設計を行います。

**3.2.1. プレイヤー制御と操作性**

*   **入力システムの改善:**
    *   **矢印キーの修正:** `index.html`の1216-1219行目にあるキーボード入力の反転ロジックを修正し、標準的なWASD/矢印キー操作（W/↑で前進、S/↓で後退、A/←で左移動、D/→で右移動）に合わせます。
    *   **エイムシステム:**
        *   **ソフトロックオン:** 近くの敵を自動的にターゲットし、攻撃が敵に向かうように補正するシステム。ユーザーはターゲットを切り替えることができる。
        *   **フリーエイム:** マウス（PC）または右スティック（モバイル）でカメラと照準を自由に操作し、攻撃方向を決定する。弓矢や一部のスキルで活用。
        *   **攻撃の多様化:** 各形態で異なる攻撃方法（近接、遠隔、範囲攻撃など）を実装し、エイムの必要性を高めます。
    *   **モバイル操作の最適化:** 仮想ジョイスティックとボタンの配置、サイズ、フィードバックを調整し、より快適な操作感を提供します。特に、カメラ操作の感度調整機能を追加します。
*   **カメラシステム:**
    *   **追従カメラ:** プレイヤーの動きに合わせてスムーズに追従する三人称視点カメラ。障害物による視界の遮断を避けるためのコリジョン検出と補正。
    *   **エイムモードカメラ:** エイム時には、より精密な操作を可能にするために、カメラがプレイヤーに近づいたり、肩越し視点に切り替わったりする機能を導入します。

**3.2.2. 変身システムと形態別能力**

*   **変身チェーンの深化:** GDDの変身チェーン（ネズミ→キツネ→ヘビ→ワシ→オオカミ→原始人→???）を核とします。
*   **重ね着式変身の活用:** 下位形態が上位形態では解けない謎の鍵となるというコンセプトを強化します。
    *   例: ネズミ形態でしか入れない狭い通路、ヘビ形態でしか登れない壁、ワシ形態でしか到達できない高所など、各形態の特性を活かしたパズル要素や探索要素をステージデザインに組み込みます。
*   **本能ゲージ (Instinct Gauge):**
    *   **回復メカニクス:** 「その形態らしい行動」での回復を具体化します。例：ネズミで隠密行動、ワシで空中捕食、オオカミで群れを指揮など。これにより、プレイヤーは各形態のプレイスタイルを自然と学ぶことができます。
    *   **消費メカニクス:** 変身、特殊スキル、強力な攻撃などに本能ゲージを消費させ、戦略的なゲージ管理を促します。
*   **感覚システム (Sense Mode):**
    *   **視覚表現の強化:** GDDにある「形態別知覚」をThree.jsのポストエフェクトやシェーダーで視覚的に表現します。例：
        *   **ネズミ:** 紫外線視点（青紫のフィルター、隠れた通路や敵の痕跡が強調される）。
        *   **ヘビ:** 熱源探知（赤外線サーモグラフィーのような視界、暗闇の敵や隠れたオブジェクトが赤く表示される）。
        *   **ワシ:** 超高解像・高コントラスト（遠方のオブジェクトが鮮明に見え、広域マップが自動的に展開される）。
    *   **聴覚表現:** 各形態で聞こえる音の範囲や種類が変わる（例：ネズミは微細な足音、オオカミは遠吠えの反響）。

**3.2.3. 戦闘システム**

*   **弱点×形態の掛け算バトル:**
    *   **敵の弱点:** 各敵（特にボス）は特定の形態の攻撃や能力に対して脆弱性を持つように設計します。例：
        *   飛行する敵にはワシの空中攻撃が有効。
        *   硬い装甲を持つ敵にはオオカミの群れ攻撃で連携して破壊。
        *   素早い敵にはキツネの素早さで追いつき、カウンター。
        *   隠密行動が必要な敵にはネズミで奇襲。
    *   **パズル的要素:** 敵の行動パターンを観察し、適切な形態に切り替えて弱点を突く戦略的な戦闘を重視します。
*   **アクションの多様化:**
    *   **通常攻撃:** 各形態で異なるモーションとヒットボックスを持つ通常攻撃。
    *   **特殊スキル:** 各形態固有のアクティブスキル（例：ネズミの「隠密ダッシュ」、キツネの「跳躍攻撃」、ヘビの「巻きつき拘束」、ワシの「急降下攻撃」、オオカミの「咆哮バフ」）。
    *   **回避行動:** ダッシュ、ローリング、空中回避など、各形態に合わせた回避アクション。
*   **ダメージ表現とフィードバック:**
    *   ヒットストップ、画面揺れ、パーティクルエフェクト、サウンドエフェクトを強化し、攻撃のヒット感を向上させます。
    *   敵のHPバーや弱点部位の視覚的表示を明確にします。

**3.2.4. 生態系AIとワールドインタラクション**

*   **食物連鎖AIシミュレーションの拡張:**
    *   GDDにある「プレイヤーがいない場所でも生態系が動いている」というコンセプトを実装します。NPC動物が捕食・被食の関係で行動し、個体数が増減するシミュレーション。
    *   プレイヤーの行動（特定の動物を大量に狩る、縄張りを広げるなど）が生態系に影響を与え、ステージの状況が変化するダイナミックなワールドを構築します。
*   **仲間システムと縄張りネットワーク:**
    *   **仲間の獲得:** 倒した敵の一部が仲間になるだけでなく、特定のクエストをクリアしたり、特定の行動（例：傷ついた動物を助ける）によって仲間が増えるシステム。
    *   **群れコマンド:** オオカミ形態で解放される「群れコマンド」を強化し、仲間に指示を出して戦闘やパズルを解決できるようにします。
    *   **縄張りビジュアライズ:** GDDにある「根のような縄張り模様」をThree.jsのシェーダーでリアルタイムに表現し、HUDの代替として機能させます。縄張りが侵食されるアニメーションも実装します。

**3.2.5. ステージ構成とナラティブ**

*   **GDDステージの深化:** GDDに記載された各ステージ（草原の夜明け、湿地のヘビ王国、断崖のワシの縄張り、森のオオカミ群れ、人間の村、文明の心臓部）のコンセプトを基に、より詳細なレベルデザインを行います。
    *   各ステージで新しい形態のアンロック、新しいメカニクスの導入、物語の進行を密接に連携させます。
*   **記憶の断片 (Memory Shard):**
    *   収集要素としてだけでなく、集めることで物語の背景、生態系破壊の原因、各動物の視点からの世界観が明らかになるような、意味のあるナラティブ要素として機能させます。
    *   UIで収集した断片を閲覧できる「図鑑」のような機能を実装します。
*   **プレイヤー感情曲線:**
    *   各ステージのイベント、敵の配置、BGM、視覚演出などを通じて、GDDにある「恐怖 → 解放感 → 全能感 → 重さ・不安 → 使命感」という感情変化をプレイヤーに体験させます。
    *   特に「人間の村」での選択と道徳的ジレンマは、エンディング分岐に直結する重要な要素として、プレイヤーに重い決断を迫る演出を強化します。

### 3.3. グラフィックと演出設計

Three.jsの機能を最大限に活用し、GDDにあるビジュアル・演出設計を高品質で実現します。

*   **変身エフェクト:** GDDの「骨格リモデリング」をThree.jsのシェーダーとアニメーションで表現します。モーフィングターゲットやボーンアニメーションを駆使し、滑らかで説得力のある変身を実現します。
*   **形態別世界色調 (Color Grading):** ポストエフェクトとして実装し、各形態で異なるカラーフィルターやトーンマッピングを適用します。これにより、視覚的な没入感を高めます。
*   **生態系粒子シミュレーション:** Three.jsのパーティクルシステムを強化し、空気中の胞子、花粉、水分子などをリアルタイムでシミュレーションします。形態によって見え方が変わる（例：ワシは上昇気流、ネズミは土埃）演出を実装します。
*   **時間軸エフェクト:** リアルタイムの昼夜サイクルを実装し、太陽と月の位置、影の長さ、環境光の変化を動的に表現します。各形態の得意な時間帯をゲームプレイに組み込みます。
*   **毛並み・鱗シェーダー:** 各クリーチャーモデルに高品質な毛並みや鱗のシェーダーを適用し、よりリアルな質感を実現します。風による揺れや光の反射を表現します。
*   **ハプティクスフィードバック:** モバイル版では、GDDにある「触れた際のハプティクス（スマホ振動）」を実装し、攻撃ヒット、ダメージ、変身などの重要なイベントで振動フィードバックを提供します。

### 3.4. UI/UX設計

GDDのUI/UX設計方針を基に、視認性と操作性を両立したUIを実装します。

*   **HPゲージ/本能ゲージ:**
    *   GDDの「画面端の毛並み・鱗の状態」「キャラクターの目の発光色」という表現を維持しつつ、視認性を高めるための微調整を行います。
    *   ダメージを受けた際の毛並みの逆立ち、本能ゲージが減少した際の目の輝きの変化などをより明確に表現します。
*   **ミニマップ:**
    *   ワシ形態でのみ俯瞰マップが展開されるというコンセプトを維持します。他の形態では、周囲の状況を把握するための「感覚システム」をより重視させます。
    *   ミニマップ上には、目標、敵の位置、仲間の位置などを分かりやすく表示します。
*   **パネルUI:** スキルパネル、インベントリパネル、マップパネルなどを実装し、情報を整理して表示します。PCではキーボードショートカット、モバイルではタッチ操作に最適化します。
*   **トースト通知:** 重要な情報（アイテム獲得、スキル習得、クエスト更新など）は、画面上部に一時的に表示されるトースト通知でプレイヤーに伝えます。

## 4. 開発プロセスとGitHub連携

*   **GitHubリポジトリ:** ユーザー指定の`guttyanneruuuuuu/gjm`リポジトリを継続して使用します。
*   **ブランチ戦略:** `main`ブランチは常に安定版を保ち、機能追加やバグ修正は`feature/`や`bugfix/`ブランチで開発し、プルリクエストを通じて`main`にマージします。
*   **コミットメッセージ:** 変更内容が明確にわかるようなコミットメッセージを記述し、履歴の追跡を容易にします。
*   **逐次プッシュ:** ユーザーの要望通り、開発の進捗に合わせて定期的にGitHubにプッシュします。
*   **最終成果物:** 最終的には、ビルドシステムを用いてすべてのコードとアセットをバンドルし、単一の`index.html`ファイルとして出力します。
